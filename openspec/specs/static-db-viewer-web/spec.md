# Static DB viewer web

## ADDED Requirements

### Requirement: Static DB viewer entry point
The system SHALL provide a static web entry point that can be served from a folder (for example `docs/static-db-viewer/`) without any backend logic.

#### Scenario: Open viewer from GitHub Pages
- **WHEN** a user opens the static viewer URL hosted on GitHub Pages
- **THEN** the browser SHALL load the entry HTML file for the viewer
- **AND THEN** the viewer SHALL be fully functional using only static assets from the same site

### Requirement: Agenda selection and navigation
The static viewer web SHALL allow the user to select any of the three agendas (ODS, Agenda Urbana, descriptive Agenda Urbana) and navigate their available indicators in a human-friendly way.

#### Scenario: User selects an agenda
- **WHEN** the user selects an agenda (ODS, AU or descriptive AU) in the viewer
- **THEN** the viewer SHALL display the hierarchical structure of that agenda (for example, objectives, metas or equivalent levels)
- **AND THEN** the viewer SHALL list the indicators associated with the selected part of the hierarchy

### Requirement: Municipal selection within the viewer
The static viewer web SHALL allow the user to select a municipality from the configured sample and see which indicators are available for that municipality in each agenda.

#### Scenario: User selects a municipality
- **WHEN** the user selects a municipality from the list of available cities in the viewer
- **THEN** the viewer SHALL highlight or list the indicators that have values for that municipality in the currently selected agenda
- **AND THEN** the viewer SHALL show the municipality name clearly so that the user knows which city is being inspected

### Requirement: Hierarchical display of indicators
The static viewer web SHALL display indicators within each agenda in a hierarchical structure that is understandable to humans, rather than only using raw IDs.

#### Scenario: View hierarchical indicators for an agenda
- **WHEN** the user is viewing a particular agenda in the viewer
- **THEN** the viewer SHALL render indicators grouped by their parent levels (for example, objectives and metas for ODS or general and specific objectives for AU)
- **AND THEN** each indicator entry SHALL include at least a human-readable name and its identifier and the value of the indicator for the currently selected municipality (empty if no value is available or no municipality is selected)

### Requirement: Static data loading
The static viewer web SHALL load its data about agendas, indicators and municipalities exclusively from static JSON files generated by the pipeline.

#### Scenario: Viewer loads data from static JSON
- **WHEN** the viewer is loaded in a browser
- **THEN** it SHALL fetch one or more JSON files (for example `data/agendas.json` and `data/municipios.json`) from its own static folder
- **AND THEN** it SHALL render the UI based on the contents of those files without requiring any dynamic API calls
